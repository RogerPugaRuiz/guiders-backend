## 🔧 Lo que fue difícil

1. **DTO Update validators misplaced (Story 1.1)**
   - En el code review, el `@Matches(ORIGIN_REGEX)`, `@ArrayMaxSize(50)`, etc. quedaron en `BrandingDto` en lugar de `UpdateWhiteLabelConfigDto` (mi `oldString` matcheó contra el archivo equivocado en 2 edits).
   - **Lección**: cuando un `oldString` matchea múltiples bloques en un archivo, agregar más contexto único o reescribir el archivo completo.
   - **Resolución**: detectado en code review, fixed antes de merge.

2. **`EmbedTokenGuard` F2 — class-level guard bloquea refresh**
   - El `IntegrationApiKeyGuard` global aplicaba a `/v2/integration/embed/start` Y `/v2/integration/embed/refresh`. El iframe nunca envía `x-api-key` → endpoint inaccesible en producción.
   - **Lección**: cuando agregas un nuevo endpoint a un controller con guards globales, evalúa explícitamente si el nuevo caller tiene las credenciales esperadas.
   - **Resolución**: mover el guard a nivel de método (solo `start` lo usa).

3. **`EmbedTokenRequest.embedToken` no opcional en el interface**
   - El tipo mentía sobre la opcionalidad real.
   - **Lección**: `interface X extends Request` con `field: string` no marca el campo como garantizado por el guard. Hacerlo `field?: string` y castear en el controller.
   - **Resolución**: F11 del code review de Story 1.4.

4. **`<= EmbedTokenGuard, EmbedController, ...>` no registrados en el module**
   - Sin el patch F1 del code review, Nest no habría podido instanciar el `EmbedController` con `RefreshEmbedTokenCommandHandler` y `EmbedTokenGuard` (Nest throws `UnknownDependenciesException` en bootstrap).
   - **Lección**: el code review con múltiples capas (Blind Hunter, Edge Case Hunter, Acceptance Auditor) **encontró este bug que el unit/e2e test no detectó** (porque mockean las dependencias). Valor demostrado del adversarial review.
