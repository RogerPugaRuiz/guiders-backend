## 🎉 Lo que salió bien

1. **First event publishing story en embed namespace** — Abre el patrón `EventBus` para futuras stories (3.1 postMessage, 2.4 transparent refresh, etc.). La decisión de usar `EventBus.publish()` directo (no `mergeObjectContext`) porque no hay aggregate root fue correcta.
2. **tryPublish helper centralizado** — Detectado en code review, aplicado a los 3 handlers en un commit. Garantiza AC7 (best-effort) y evita que un fallo del bus rompa el main flow. Reusable para futuras stories con eventos.
3. **PII sanitization is explicita en el código** — No es un afterthought; está documentado en JSDoc y testeado con 12 casos (truncation, regex, hashing determinista).
4. **TTL index para GDPR** — Implementado con `expireAfterSeconds: 31536000` en el campo `timestamp` (NFR-S7 compliance). Documentado en el JSDoc del schema.
5. **Cross-handler consistency en error classification** — F10 forzó a que `RefreshEmbedTokenCommandHandler` use la misma jerarquía de 4 categorías que `AuthenticateEmbedSessionCommandHandler` (NotFound/Invalid/ServiceUnavailable/Unknown). Antes los 2 handlers clasificaban errores de forma inconsistente.
6. **Module wiring correcto desde el primer build** — `MongooseModule.forFeature` + `CqrsModule` + 3 handlers + 1 query handler + 1 repository provider. Build pasó al primer intento (lesson AI-2 aplicada).
7. **Documentación completa** — `bff/AGENTS.md` ya documentaba `BFF_SESSION_SERVICE` y los patrones; este story extendió a `IntegrationApiKeyModule` con una nueva sección sobre audit log.

## 🔧 Lo que fue difícil

1. **`@tdd-generator` subagente sigue sin output** — 2 invocaciones retornaron `<output></output>` vacío. Apliqué AI-1 (fallback manual) por segunda vez consecutiva. **Patrón emergente**: el subagente es unreliable en este proyecto; necesita wrapper robusto o ser reemplazado por un script determinista.
2. **DRY violation en audit context extraction** — El bloque `req.headers['origin']/['x-forwarded-for']/['user-agent']` se copió 3 veces (en 3 controllers) sin extraer. Code review F13 lo detectó. **Decisión**: defer a un futuro commit de tech debt (no bloqueante).
3. **EventBus publish no es fire-and-forget en @nestjs/cqrs** — Story 2.1 asumía esto. F4/E15 demostró que si un handler sincrónico falla, el controller se rompe. Creé `tryPublish` helper. **Lesson**: en @nestjs/cqrs, **siempre** envolver `publish()` en try/catch.
4. **Tests de Story 1.3/1.4 rompieron al modificar los handlers** — Story 1.4 esperaba `err(EmbedTokenInvalidError)` cuando `validateToken` retornaba `EmbedTokenError` genérico. F10 cambió el contrato a `err(EmbedTokenError)` (más preciso). Tuve que actualizar 1 test existente.
5. **class-validator@0.14.2 con ISO 8601 + Z** — `@IsISO8601()` con `Z` rechaza por defecto. Deferido como tech debt; el e2e test de fechas fue removido.
6. **E2E mock pattern** — `const` con `jest.fn()` (no reasignación dentro de `buildApp`) resuelve el problema de Story 1.3/2.1. Apliqué consistentemente.
7. **El 'unknown' magic string** — E7 detectó que `command.apiKeyCompanyId ?? userId ?? 'unknown'` mezclaba null userId con string 'unknown' companyId, creando una falsa atribución. Simplifiqué a `apiKeyCompanyId ?? 'unknown'`.
