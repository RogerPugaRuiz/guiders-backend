# Seed E2E Company

CLI command that wipes both PostgreSQL and MongoDB and rebuilds a complete, reproducible E2E environment for the frontend: one company with its admin user, auto-generated API keys, and a configurable batch of fake visitors with realistic varied data.

## When to use this skill

Trigger this skill when the user requests any of:

- "seed e2e" / "seed e2e company"
- "prepara/genera/resetea el entorno e2e"
- "limpia la base de datos y crea una empresa de pruebas"
- "necesito visitantes ficticios para el frontend"
- "datos de prueba para tests end-to-end"
- "popular MongoDB/Postgres con datos de prueba"

Do **not** use this skill for: running tests, creating real companies in production, or seeding only one isolated entity (use the dedicated CLI command for that, e.g. `create-company`).

## Command location

- Source: `tools/cli.ts` (command registered as `seed-e2e-company`)
- Entry: `bin/guiders-cli.js`
- npm alias: `npm run seed:e2e`

## Quick usage

```bash
# Default: cleans DBs, creates "Guiders E2E Test Company" + 150 visitors
npm run seed:e2e

# Custom company + 500 visitors
npm run seed:e2e -- \
  --name "Mi Empresa E2E" \
  --domain e2e.miweb.local \
  --aliases "www.e2e.miweb.local,test.miweb.local" \
  --admin-email admin@miweb.local \
  --visitors 500

# Append visitors without wiping existing data
npm run seed:e2e -- --skip-clean --visitors 50

# Direct invocation (without npm wrapper)
node bin/guiders-cli.js seed-e2e-company --visitors 200
```

## CLI options

| Flag | Default | Description |
|------|---------|-------------|
| `--name <name>` | `Guiders E2E Test Company` | Company display name |
| `--domain <domain>` | `e2e.guiders.local` | Canonical site domain |
| `--aliases <list>` | `www.e2e.guiders.local,test.e2e.guiders.local` | Extra domains (comma-separated) |
| `--admin-name <name>` | `E2E Admin` | Admin full name |
| `--admin-email <email>` | `admin@e2e.guiders.local` | Admin email |
| `--admin-tel <tel>` | `+34600000000` | Admin phone |
| `--visitors <n>` | `150` | Number of fake visitors to generate |
| `--skip-clean` | `false` | Skip the database wipe step |

## What the script does (in order)

1. **Database cleanup** (unless `--skip-clean`):
   - PostgreSQL: `TRUNCATE ... CASCADE` on every public table with `SET CONSTRAINTS DEFERRED`
   - MongoDB: `deleteMany({})` on every collection
2. **Create company + admin** via `CreateCompanyWithAdminCommand` with canonical site + aliases
3. **Wait 6 seconds** so async event handlers run (notably `CreateApiKeyOnCompanyCreatedEventHandler` which auto-generates API keys per domain)
4. **Recover real IDs** generated by the system (companyId, siteId) and API keys per domain via repositories
5. **Bulk-create N visitors** using `VisitorV2.create()` directly via repository in batches of 25 (parallel) with:
   - Unique fingerprints
   - Realistic random IPs
   - 5 rotating user-agents (Chrome, Safari, Firefox, iPhone, Android)
   - 7 rotating URLs simulating navigation
   - Mixed lifecycle distribution: `ANON` / `ENGAGED` / `LEAD` / `CONVERTED` (respecting aggregate transition rules)
   - ~2% flagged as internal (commercial team)
   - GDPR consent accepted (v1.0.0)
   - Initial session created automatically by the factory
   - Domain events committed via `EventPublisher.mergeObjectContext()` + `commit()`
6. **Print summary**: companyId, siteId, domain, admin credentials, primary API key, and visitor success/failure counts

## Output you can expect

The final summary prints exactly what the frontend needs to configure its E2E test suite:

```
✅ Seed completado:
   Company ID: <uuid>

   Domain:     e2e.guiders.local
   Admin:      admin@e2e.guiders.local
```

## Prerequisites

- PostgreSQL and MongoDB running and reachable (check `.env` / `NODE_ENV=development`)
- Migrations applied: `npm run typeorm:migrate:run`
- Dependencies installed: `npm install`

## Important notes

- **Destructive by default**: wipes ALL data in both databases. Never run against production. Always use `NODE_ENV=development` (the npm script forces this).
- **The 6s wait is required**: API keys are generated by an async event handler. Reducing this can yield "no API key found" in the summary.
- **Visitor creation uses the repository directly**, bypassing `IdentifyVisitorCommand`, to avoid API key validation overhead during bulk insert.
- **Failures are silent per visitor**: if a lifecycle transition fails, the visitor is still saved in its current state and counted as success. Hard failures (save errors) increment the failed counter shown in the summary.
- **Re-running without `--skip-clean`** is the canonical way to reset a dirty environment between test runs.
